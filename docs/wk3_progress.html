这不是一个单纯的 token 仓库，而是一套以 `source/tokens.json` 为单一事实源的设计系统运行模型，用来把 AI、Figma 和 runtime 对齐到同一个 canonical model。

一、核心结论

当前阶段最重要的成果，不是多了几个 token 文件，而是建立了一套清晰的“source - adapter - consumer”关系。

以 source 为先

一个 canonical model

系统的真实含义、命名、主题映射和语义描述，都归属于 source 层。

以 adapter 驱动

不同消费者拿到不同形态

AI、Figma、runtime 都消费同一系统，但各自需要不同形态的 projection。

以角色为中心

不同角色保留原生工作方式

PM 和 DEV 主要消费 schema；UX 负责在 Figma 中确认变更，再推动 source 更新。

二、当前 Skill 构成与路线图能力

当前真正稳定公开的是 `adapt`；`audit` 与 `normalize` 仍清楚地保留为下一阶段扩展。

当前模块

adapt

  • 从 canonical source 生成 AI、Figma、runtime 三类 projection
  • 保留 `primitive / semantic / pattern / component` 的边界
  • 提供 builder scripts 与 validation 规则
未来模块

未来模块

  • 负责命名、结构、格式归一化
  • 适合承接 reverse import 或历史资产迁移
  • 重点是归一化对齐，不是 consumer projection

三、SSOT 模型与建议文件结构

SSOT 的关键，不只是“有一个 source 文件”，而是清楚地区分哪些是作者维护的 artifact，哪些是生成产物。

系统层级

Source 层

`source/tokens.json` 存储 canonical 含义、命名、主题对齐与描述，是唯一适合手动维护 token 事实的地方。

构建与校验层

builder scripts 负责 resolve alias、校验 schema、保证 Light / Dark parity，并生成不同消费者需要的 outputs。

消费层

AI、Figma、runtime 拿到各自最适合使用的 projection，但仍可追溯回 source 路径。

单向流转 
source/tokens.json
        ↓
resolve + validate + build
        ↓
adapters/ai/*
adapters/figma/*
adapters/runtime/*
        ↓
AI / Figma DS / runtime

这个方向必须保持单向。Figma 可以是评审 surface，但不能成为 source 的反向 authority。

四、核心角色、使用场景与 artifacts 映射

核心不是“谁都用同一套工具”，而是“每个角色都在自己的原生工具里工作，但围绕同一个 canonical source 协同”。

PM

探索者 / 消费者

PM 通过 AI-native schema 做方案探索与原型讨论，不直接维护 token source。

AI 工具 PRD / 文档 原型评审
ai.tokens.json index componentIndex
UX

生产者 / 维护者 / 审核者

UX 在 Figma 中确认 value 变化，再通过 AI 更新 source token，并重建 adapters 回投到 Figma DS。

Figma DS AI 辅助 source 编辑 设计评审
source/tokens.json figma adapter 治理流程
DEV

运行时消费者

DEV 在代码和 runtime pipeline 中直接消费 schema，不需要通过 Figma 维护 token。

代码仓库 主题管线 实现评审
tokens.runtime.json tokens.css.json tokens.tailwind.json
PM 主要消费的 artifact

面向 AI 的直接 schema

让 PM 能从业务意图出发，直接获得符合系统约束的 prototype 结构。

UX 主要维护的 artifact

Figma DS + source token

UX 的职责不是在 Figma 里直接改 source，而是在 Figma 中确认变更，再把 canonical change 体现在 source。

DEV 主要消费的 artifact

runtime-friendly 输出

DEV 最需要的是稳定键名、明确模式和可直接接入代码的输出。

五、首选工作流程

这部分的重点是把 authored artifact 和 derived artifact 分清楚。Figma 可以是设计侧工作层，但 source 才是系统 authority，adapter 负责把 source 投影给不同消费者。

步骤 01

在 Figma 工作层完成讨论

UX 在 Figma DS 中讨论组件、value 和视觉效果，用它做 visual review 与设计确认。

步骤 02

把设计结论转译回 canonical source

确认的是 source change，而不是让 Figma 文件直接反向改写 `source/tokens.json`。

步骤 03

通过 AI 更新 source token

AI 在 canonical schema 内协助更新 source，并保留命名、结构、主题对齐和引用关系。

步骤 04

重建 adapters 并重新同步

Figma、AI 与 runtime 侧都重新拿到更新后的 projection，避免 consumer 自己长出独立事实。

Authoritative / Derived

Source remains authoritative. Figma and runtime stay derived.

保持 authoritative 的始终是 source。 Figma、AI schema 和 runtime outputs 都是面向特定消费者的 projection。它们可以被优化为更适合使用，但不能替代 source 成为新的系统 authority。

Figma: 设计工作层 Source: 唯一 authority Adapters: 消费侧 projection

六、工作流程的视觉化展示

从 adapt 的角度看，这不是一个 “Figma 同步 token” 的流程，而是一个 “canonical source 被投影到不同消费者” 的流程。

01

设计侧工作层

在 Figma DS 中完成设计讨论、变量审视和视觉确认。

02

系统级变更决策

UX 把设计讨论沉淀为 source 层应该发生的 canonical change。

03

更新 canonical source

AI 在 canonical schema 内更新 source token，并通过 validation 保持一致性。

04

生成 consumer projections

AI、Figma 和 runtime projections 从同一次 source 更新中重新生成。

05

各消费者重新对齐

PM、UX、DEV 在各自工具里继续工作，但消费的是同一个 source 投影出来的最新结果。

七、四层 token 分类与 source token 的语义结构

这套系统的 token 不是平铺的 value 列表，而是通过四种 view 来组织同一套设计事实。

视角 01

primitive

基础视觉比例尺，例如 color、spacing、size、radius。

视角 02

semantic

界面角色语义，例如 text、background、border、fill、gradient。

视角 03

pattern

可复用结构模式，例如 surface、modal form、list item。

视角 04

component

具体组件绑定，例如 variant、state、final property。

结构示意 
primitive.color.blue.500
        ↓
semantic.text.action
        ↘
pattern.modal.form.title
component.button.primary.default.text

`pattern` 与 `component` 是并列的上层消费视角。

source token 的关键字段
  • `$type`：定义 token contract
  • `$value`：保留 raw value 或 alias graph
  • `$description`：给 AI 和治理流程提供语义依据
  • Light / Dark parity：保证双主题路径对齐
  • alias traceability：让 semantic / component 能回溯到 primitive
source 语义示例 
semantic.text.action
  $type: color
  $value: {primitive.color.blue.500}
  $description: "Interactive action text..."

这也是为什么 source 不只是存视觉值，而是在存“意义”和“使用理由”。这层结构对于 AI reasoning 尤其重要。

八、token schema 类型、source token 语义解读与 Figma schema 解读

source schema 和 Figma schema 不是相互替代关系，而是 authority 与 projection 的关系。

当前 source 的整体结构
$metadata $themes Primitive Light Dark
Light.semantic Light.pattern Light.component Dark.semantic Dark.pattern Dark.component

当前 source 已经具备四层视角和双主题投影。

不同 adapter 的类型
  • AI adapter：额外提供 `index` 与 `componentIndex`，便于 AI 推理与组件重建
  • Runtime adapter：输出 runtime JSON、CSS variables、Tailwind、React、mobile 等实现友好形态
  • Figma adapter：按 collection / mode 组织路径，以适配设计工具的导入与变量体系
AI adapter 的价值 
tokens
index
componentIndex

AI 不只需要 token tree，还需要 resolved value、references、description 和组件级索引。

Figma projection 的当前形态 
primitive
semantic/light
semantic/dark
pattern/light
pattern/dark
component/light
component/dark

这不是在“重写 source”，而是在把 source 映射成 Figma 更容易消费的形态。

source token 语义结构的意义
  • `$description` 让 token 能被 AI 理解“为什么该用它”
  • alias 关系保证 semantic / pattern / component 对 primitive 的追溯能力
  • Light / Dark path parity 让双主题结构可被稳定验证
  • pattern 与 component 的并列存在，支持“结构复用”和“组件绑定”两种表达
Figma schema 的采用原则
  • Figma 不是 source of truth，但必须是 UX 工作流中的关键工作层
  • Figma projection 应优先服务 variable grouping、mode parity 与设计师可理解性
  • 不把 AI-only metadata 强行塞进 Figma schema
  • Figma 侧的结构应保持为 source 的 projection

九、下一阶段建议

下一阶段最有价值的方向，不是继续堆更多产物文件，而是把现有 source-first operating model 补齐成更明确的治理系统。

语义深度

补足 source metadata

在 source 层增加适度的结构化语义字段，先验证是否真的提升 AI 输出质量。

系统模块

把 normalize 与 audit 正式化

让 migration、reverse import、drift detection 和治理审计拥有清晰模块边界。

流程落地

强化角色协同叙事

持续强化“在 Figma 中讨论，在 source 中定案，再向外投影”的操作习惯。
