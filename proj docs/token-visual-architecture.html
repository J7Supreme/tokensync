2. Canonical Layer Stack

这是 PRD 的完整目标模型，也是最适合被 AI 和设计系统共同理解的层级组织方式。

Implemented

Primitive

纯视觉原子材料，不带 UI 语义。当前已实现 29 个 color token。

primitive.color.blue.500

Implemented

Semantic

把视觉材料翻译成界面角色语义。当前 Light / Dark 各 24 个，全部引用 primitive。

semantic.text.primary / semantic.background.card

PRD Defined

Pattern

把多个 semantic 组织成可复用界面结构，如 surface、layout、container。

pattern.surface.card

PRD Defined

Component

把 Pattern 和 Semantic 落到组件消费规则，支持状态、variant 和运行时直接消费。

component.button.primary.default.background

为什么这套层次对 AI 更友好

  • Primitive 让模型知道可用材料边界，而不是任意造值。
  • Semantic 让模型理解“用途”，而不是只识别颜色。
  • Pattern 让模型学会常见 UI 组合关系。
  • Component 让模型直接输出可落地的按钮、卡片、输入框规则。
当前 source/tokens.json 已经完成了 “材料层 + 语义层 + mode 机制”。
一旦补齐 pattern 和 component，AI 就能从“识别设计语言”进一步升级为“重建设计系统组件”。

3. DS 原子 / 分子 / 组件映射

如果用设计系统和 Atomic Design 的语言来解释，这套 token 架构天然就是“由小到大”的可组合结构。

Token Layer DS / Atomic Design 映射 系统意义 AI 可理解的内容
primitive 视觉原子 / 材料库 建立颜色、间距、圆角、阴影等基础尺度 知道“允许使用什么值”
semantic 语义原子 / 功能角色 把视觉值映射为 text、background、border、fill 等用途 知道“什么值用于什么角色”
pattern 分子 / Pattern 结构 把多个语义 token 组合成 card、modal、layout 之类的交互片段 知道“哪些语义通常一起出现”
component 组件 / Organism / 实例规则 直达按钮、输入框、卡片在不同状态下的最终消费属性 知道“如何稳定生成组件”

4. 当前 Source 已体现的 AI Native 特性

即便当前还没补齐 component 层，现有 tokens.json 已经具备非常明确的 AI-native 基因。

语义路径可推理

semantic.text.primary、semantic.background.modal 这类命名天然可被 LLM 解析为界面角色，不是匿名样式表。

$description 可解释

PRD 要求保留非空 $description。这让 AI 不只知道 token 名，还知道使用语境、层级和意图。

双 Mode 对称

Light / Dark 路径完全一致，AI 可以基于同一语义键在不同主题下切换结果，而不是重新学习一套命名。

Alias 而非硬编码

Semantic 层通过 alias 指向 Primitive，形成“语义到材料”的可追踪链路，避免 AI 学到散乱的硬编码值。

当前已实现

Primitive: 29 color tokens

Light semantic: 20 color + 4 gradient

Dark semantic: 20 color + 4 gradient

主题机制

$themes 已定义 Light / Dark。

$metadata.tokenSetOrder 已明确 token set 顺序。

当前缺口

PRD 要求 Light / Dark 同时包含 semantic 与 component，但当前 Source 尚未加入 component。

5. 与业界标准和规范的符合点

这里的“符合”不是只指字段长得像标准，而是结构、约束、消费链路都接近成熟 token 体系的工程实践。

规范方向 当前 / PRD 对应点 意义
W3C DTCG 风格字段 使用 $type、$value、$description、theme set、metadata 便于编译器、插件、转换脚本一致处理
Alias 驱动 Semantic 必须引用 {primitive.*}，Component 优先引用 {semantic.*} 形成清晰依赖图，减少硬编码漂移
Theme parity Light / Dark 的 semantic 与 component 路径要求完全一一对应 可做自动校验、主题切换与差异比对
Adapter architecture Source 语义层与 Figma / Runtime / AI 格式层分离 符合多端设计令牌的主流编译思路
Traceability PRD 要求 Runtime 保留 sourceRef 保证从 UI 输出反查到 token 源头
Gradient special handling Gradient 作为独立类型处理，并在 Figma 侧走 Paint Styles 符合工具能力边界，避免把不兼容能力硬塞进 Variables
